Purpose

Why PhishTank exists

Modern phishing is no longer only a suspicious message body. Attackers distribute evidence across headers, hidden links, visual attachments, multilingual text, and QR codes to bypass single-layer filters.

⚠

The problem

  • Body-only detection can miss malicious URLs, spoofed headers, or visual attachments.
  • URL-only detection misses social engineering and legitimate-looking but dangerous messages.
  • Arabic and multilingual emails need translation-aware processing.
  • QR codes and PDF/image attachments can hide the actual payload.
🛡

The solution

  • Extract clean, structured evidence from every email through IMAP, MIME parsing, OCR, and QR decoding.
  • Route each evidence type to the correct model instead of sending everything everywhere.
  • Fuse model probabilities so one dangerous attachment or link can raise the final risk.
  • Show the admin final verdicts, component scores, evidence traceability, and system health.
Architecture

Collect → Extract → Route → Analyze → Fuse → Decide

The system is modular and can run across devices. The parser/OCR device creates the evidence JSON, while model services return probabilities to the dashboard and fusion layer.

1

External IMAP

Raw mailbox source for incoming messages and attachments.

2

Parser + OCR + QR

MIME parsing, visible text cleanup, PDF/image OCR, embedded PDF links, and QR decoding.

3

Evidence JSON

Clean contract containing headers, body text, body URLs, OCR text, PDF links, and QR links.

4

Dashboard + Fusion

Routes evidence, calls model APIs, collects scores, and calculates final probability.

5

Final Decision

Safe, Suspicious, or Malicious with pass, review, or block workflow.

Model layer

Specialized modules for specialized evidence

Each model receives only the evidence it understands. This improves traceability and prevents one blind spot from controlling the whole verdict.

Body / NLP

BERT semantic analysis

Fine-tuned BERT analyzes body text and OCR text for phishing intent, urgency, impersonation, and suspicious wording.

  • Scans body and OCR separately
  • LIME explainability support
  • Visible-text cleanup for HTML emails
Arabic-aware

NLLB-200 translation

Arabic and other supported non-English messages are translated to English before BERT classification while preserving URLs and emails.

  • Language detection
  • Line-by-line translation
  • Security token preservation
Header

Metadata anomaly detection

Header analysis studies SPF, DKIM, DMARC, Return-Path, routing, malformed fields, and sender-domain consistency.

  • Language-independent evidence
  • Random Forest / Stacking experiments
  • FastAPI model endpoint
URL

Static link risk scoring

The URL service analyzes lexical and structural URL indicators without relying on webpage content execution.

  • Body, PDF, OCR, and QR links
  • Logistic Regression + SVC + Decision Tree
  • Per-link phishing score
OCR

DeepSeek-OCR-2 attachments

Images and PDF pages are converted to OCR-ready images so hidden visual text becomes model-readable evidence.

  • PNG, JPG, JPEG, PDF
  • Clean Markdown output
  • Attachment provenance
QR / Quishing

QR payload extraction

OpenCV QR decoding reveals QR URLs hidden in images or rasterized PDF pages, then routes them to URL analysis.

  • Single and multi-code decoding
  • Duplicate removal
  • URL-only filtering
Parser

MIME and HTML cleanup

The parser separates body text, headers, URLs, attachments, embedded PDF links, and HTML anchor targets.

  • Preserves href targets
  • Does not mix attachments into body
  • Outputs one JSON per email
Dashboard

Security operations view

The admin dashboard syncs parser output, calls model APIs, applies fusion, and presents final decisions and evidence.

  • Counts and trend charts
  • Health checks
  • Pass / block workflow support
Fusion layer

Weighted probabilistic decision

P_text = max(p_body, p_ocr1, ..., p_ocrk) P_url = max(p_url1, ..., p_urln) P_header = p_header P_final = 0.40P_text + 0.40P_url + 0.20P_header

Max aggregation prevents one risky link or attachment from being diluted by many clean sections.

SAFE
Final score < 0.40Allow or monitor normally.
SUSPICIOUS
0.40 ≤ final score < 0.70Review before delivery or allow manually.
MALICIOUS
Final score ≥ 0.70Block, quarantine, or alert according to policy.
